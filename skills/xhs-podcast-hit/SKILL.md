# XHS Podcast Hit

执行目标：`用户提出需求 -> 获取数据 -> 创作爆款`。

## 执行步骤

1. 识别意图（新增显式参数）
- 必问或自动解析三件事：
  - 是否指定播客名（可选，不输入则自动挑）
  - 是否指定单集名（可选，不输入则自动挑）
  - 需要输出几个推荐（默认 1）
- 抽取：主题、情绪诉求（缓解焦虑/搞钱/成长）、数量。
- **强制前置澄清规则**：如果用户只给了主题/情绪/数量，而**没有明确回答“是否有必推播客、是否有必推单集”**，必须先补问，不能直接检索或直接出文案。
- 推荐的最小澄清问法固定为：
  1) 有没有**必推播客**？没有就回“无”
  2) 有没有**必推单集**？没有就回“无”
  3) 要几个推荐？
- 只有在用户明确回答了上述 3 项（哪怕回答“无”），才能进入数据库检索与写作。

2. 检索数据库（必须先检索再创作）
- 运行：
  - 默认：`python3 /Users/look/openclaw/huanglei/workspace/skills/xhs-podcast-hit/scripts/query_podcast.py --db /Users/look/openclaw/huanglei/workspace/app.db --topic "<用户主题>" --limit <n>`
  - 指定播客：追加 `--podcast "<播客名>"`
  - 指定单集：追加 `--episode "<单集名>"`
- 若结果为空：降级为更宽泛关键词再查一次（去掉修饰词，仅保留核心词）。
- 按评分机制自动排序，取 Top N。

3. 评分机制（必须展示）
- 公式：`Score = 50 * 简介语义相似度 + 20 * 关键词命中率 + 30 * 小宇宙热度权重`
- 热度权重分层：S / A / B（基于订阅量动态分位）
  - S = 1.0
  - A = 0.7
  - B = 0.4
- 输出每个候选的：`score / semantic_similarity / keyword_hit_rate / heat_tier`。

4. 去重与排位规则（必须执行）
- 同一个播客在最终输出里最多出现 **1 次**（一个播客只保留评分最高的单集）。
- 若用户指定了必选播客：
  - 必选播客只出现一次。
  - 必选播客排在第 2 或第 3（默认第2，用户可改）。
- 除必选播客外，其余位次按评分从高到低补齐。
3. 按模板生成文案
- 使用 `references/prompt-template.md` 的结构和语气。
- 强制要求：
  - 先写开场共鸣段，再输出推荐列表
  - 每条推荐固定 3 行：`🎧播客名` + `推荐单集` + `🌟情绪化点评`
  - 不输出“热度背书”单独行
  - 标签放在最后，5-10 个
  - 高频 emoji，但不过度堆叠

4. 输出质量闸门
- 检查是否真的引用了数据库字段（不要编造单集信息）。
- 点评必须和 episode_desc 有语义关联，禁止纯空想。
- 文案保持“像人写的”，避免模板腔。

## 输出格式

[共鸣引入 2-3 句]

🎧 [播客名A]
推荐单集：[单集标题]
🌟 [点评]

🎧 [播客名B]
推荐单集：[单集标题]
🌟 [点评]

...（共 N 条）

#标签1 #标签2 #标签3 #标签4 #标签5

## 调试模式（开发时）
若用户说“显示检索依据”，先输出：
- 命中关键词
- Top 候选（播客名/单集名/热度字段）
然后再输出正文。

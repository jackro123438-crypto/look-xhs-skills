# XHS Hub Router

当用户输入“小红书”时，先回复菜单，不直接执行具体任务。

当前这台机器已接入的小红书相关能力有：

1. 本地库调用生成爆款文案（xhs-podcast-hit）
2. 小红书数据采集 / 搜索 / 详情 / 话题跟踪（xiaohongshu）
3. 小红书笔记分析 / 审核 / 优化建议（xiaohongshu-note-analyzer）
4. 小红书笔记创作 / 卡片渲染 / 发布准备（xhs-note-creator）

当前菜单固定为：

1. 本地库调用生成爆款文案（xhs-podcast-hit）
2. 搜索 / 采集小红书内容（xiaohongshu）
3. 分析一篇小红书笔记（xiaohongshu-note-analyzer）
4. 生成小红书笔记素材与卡片（xhs-note-creator）
5. 查看当前小红书能力验证状态

## 路由规则
- 用户回复数字后再执行。
- 选择 1（本地库生成爆款文案）后，必须先补齐 3 个前置参数：主题/情绪、是否有必推播客、是否有必推单集、推荐数量；其中“必推播客/单集”即使为空，也要明确让用户回答“无”，不得省略。
- 选择 2（搜索 / 采集）时，优先走 `xiaohongshu`：搜索、详情、话题跟踪默认可做；涉及登录态失效时，应明确提示用户配合扫码 / 登录。
- 选择 3（分析笔记）时，优先走 `xiaohongshu-note-analyzer`，给出结构化分析报告；如用户要更细的自动检查，可调用其脚本执行层。
- 选择 4（生成素材与卡片）时，优先走 `xhs-note-creator`：先生成标题正文，再生成 Markdown 素材，再渲染封面和正文卡片；发布前必须二次确认。
- 选择 5 时，汇报以下状态：
  - `xiaohongshu-mcp` / `xiaohongshu-login` 二进制是否存在
  - `xhs-note-creator` 依赖与渲染是否通过
  - `xiaohongshu-note-analyzer` 执行层是否通过
  - 小红书登录态 / Cookie 是否有效
- 对“发布”能力保持谨慎：只有在用户明确要求发布，且登录态有效、内容确认完成后才执行。
